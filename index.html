As quatro squads

Agrupadas por natureza do trabalho, não por feature isolada. As Squads 1-4 constroem (Build, Scrum); a Squad 5 garante que o que já está em produção continue funcionando (Run, Kanban).

Design system é definido por esta squad e consumido — não recriado — pelas demais.

Operação

A quinta squad — Run

Diferente das demais (que constroem), esta squad garante que clientes já integrados continuem funcionando. Modelo operacional, não de projeto — fluxo contínuo via Kanban, não sprints.

05
Run — Operação & Sucesso do Cliente Integrado
Garante o pleno funcionamento de todo cliente já integrado, em qualquer produto — CaaS, PIX, Seguros, Consórcio, BaaS+IA e Débito Veicular. É o "Run it" do Build it and Run it.
8 pessoas 5 capabilities 2 riscos Kanban
Ver capabilities, riscos e premissas ▾
Capabilities
  • Monitoramento transversal de todos os produtos em produção
  • Triagem e resolução de incidentes (Status Page, alertas, dead-letter queue de webhooks)
  • Suporte a parceiros integrados — N1/N2, escalação para squads de Build quando necessário
  • Gestão de rate limit e cota por cliente em produção real
  • Acompanhamento de SLA e relacionamento pós-integração (não pós-venda — pós-integração técnica)
Por que Kanban, não Scrum
  • O trabalho chega de forma imprevisível (incidente, ticket, alerta) — não cabe em backlog priorizado por sprint
  • Métrica de sucesso é tempo de resolução (lead time), não velocidade de entrega de feature
  • Fluxo contínuo: a squad nunca "termina uma sprint" — ela sustenta uma operação 24/7
Perfis (8)
  • 1 PM (ou Service Owner) · 3 engenheiros de suporte/SRE (plantão revezado) · 2 especialistas de produto (cobrindo os 6 produtos em dupla) · 1 analista de observabilidade · 1 tech lead de operação
Riscos
Cobrir 6 produtos diferentes com 8 pessoas exige rodízio de conhecimento bem documentado — risco de dependência de pessoa-chave (bus factor) se o conhecimento não for distribuído.
Interface entre Run e as squads de Build precisa de protocolo claro de escalação — sem isso, incidentes complexos ficam "jogados" entre squads sem dono.
Premissas
Observabilidade real (Dynatrace ou equivalente) está disponível — sem isso, a squad de Run não tem visibilidade real do que está acontecendo em produção.
Cada squad de Build documenta um runbook de troubleshooting para suas funcionalidades antes do go-live — Run não deveria precisar abrir o código-fonte para diagnosticar um incidente comum.

Argumento estratégico

Por que isso importa além da tecnologia

Esta proposta não é só evolução de produto — é uma mudança de postura frente ao vendor lock-in que hoje limita o ritmo de inovação do portal.

⚡ Autonomia de plataforma

Hoje dependemos da Axway para evoluir o portal. Essa proposta nos tira dessa dependência.

Toda mudança de arquitetura, todo produto novo, toda integração com IA — hoje passa pelo roadmap e pelos prazos de um fornecedor externo. As capabilities construídas neste protótipo (MCP Server, gRPC multi-UF, Sandbox sintético, SDKs próprios) rodam em arquitetura sob nosso controle, não amarrada a licenciamento ou ciclo de release de terceiros.

Hoje — com Axway
  • Roadmap de evolução depende do vendor
  • Customização limitada ao que o produto permite
  • Licenciamento recorrente, custo cresce com volume
  • Recursos de IA (MCP) fora do escopo do produto
Com a proposta
  • Roadmap controlado internamente, sprint a sprint
  • Arquitetura própria, sem limite de customização
  • Custo de infra previsível, sem licenciamento por volume
  • MCP, gRPC e IA nativos — não dependem de roadmap externo

Atenção do programa

Riscos transversais

Afetam mais de uma squad — não pertencem a nenhuma isoladamente.

Specs desatualizadas ou inexistentes

Atravessa as Squads 2, 3 e 4 — todas dependem de specs reais dos times de produto. Recomenda-se um gate de "spec congelada" antes de cada squad iniciar a documentação.

Chave de API de LLM exposta

Se o chatbot da Squad 3 for promovido de simulado para real sem o backend-proxy já documentado no Guia Técnico, há risco real de vazamento de credencial.

Divergência doc × API real

Sem o gate de CI de /diff já especificado para o Postman Hub, qualquer squad pode publicar documentação desalinhada da API em produção.

Interface Build × Run mal definida

Sem um protocolo claro de escalação e runbooks obrigatórios antes do go-live, incidentes complexos podem ficar sem dono entre a Squad 5 (Run) e as 4 squads de Build — risco que cresce conforme mais produtos entram em produção.
